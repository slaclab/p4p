CRITICAL difference for onGet fallback vs put/rpc:
- put AttributeError → `op.done(error="Put not supported")` — this is CORRECT for put

```python
    def onGet(self, pv, op):
        """
        Called each time a client issues a Get operation on this Channel.

        The default implementation returns the current PV value.
        Override to implement demand-driven hardware reads.

        :param SharedPV pv: The :py:class:`SharedPV` which this Handler is associated with.
        :param ServerOperation op: The operation being initiated.
                                   Call ``op.done(value=v)`` to return a value,
                                   or ``op.done(error='msg')`` to signal failure.
                                   Do NOT call ``op.value()`` — GET operations carry no input value.
        """
        op.done(value=pv.current())
```

**Addition 2: _WrapHandler.onGet method** — insert after the `rpc` method in `_WrapHandler` (after line 239) and before the closing of the `_WrapHandler` class:

```python
        def onGet(self, op):
            _log.debug('GET %s %s', self._pv, op)
            try:
                M = self._real.onGet
            except AttributeError:
                # Handler has no onGet — replicate PVXS default GET: return current value
                op.done(value=self._pv.current())
                return
            self._pv._exec(op, M, self._pv, op)
```

Note: `op` here is the raw `ServerOperation` (not `ServOpWrap`) — consistent with how `rpc` passes `op` directly. The user's `onGet(pv, op)` receives the raw ServerOperation.
